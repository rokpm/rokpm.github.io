PC上提高工作效率的软件

发表于 2014-02-11   |   分类于 个人提升

作为一个对电脑各种折腾的人，市面上常见与不常见的软件都下载试用过，下面来推荐几款PC上的软件，希望这些能提高我们的工作效率。

1、Evernote

云笔记工具，有什么好的想法立马记录下来，电脑、平板、手机端都自动同步。微信、ZAKER中的文章都能直接保存到Evernote,更厉害的是绑定微博后，在微博中看到好的资讯，评论中@我的印象笔记就能直接把该条微博保存在云端。

2、Dropbox

国外的一款云盘工具。其实一般来说用国内的网盘，例如360云盘，百度云盘等比较好，虽然无法保证资料的安全性但胜在同步稳定。而我用这个主要是公司的文档都保存在上面，还有强大的分享协作功能，更重要的是安安静静，从来不打扰用户，让它自己在后台运行就行。

3、Desktops

这是最近发现的一款虚拟桌面软件。建立多个桌面按Alt+数字键进行切换。在第一个虚拟桌面上放QQ、浏览器等工作之外的东西，第二个桌面完全放工作的内容，这样当认真工作的时候就不会被其它人打扰。

4、Everthing

强大的搜索器，我们都知道Windows在Windows 8 之前的垃圾搜索功能，安装这个不到1M的软件，就可以快速对电脑硬盘中的任何内容进行搜索，完全秒杀系统自带的搜索，适合电脑中资料较多的同学，绝对神器。

5、Chrome

这个不用说了，Google出品，必输精品，虽然有内存占用太大的缺点，但简洁的界面和强大的插件觉得值得尝试，比如使用插件Adblock，就可以屏蔽网页上的所有广告。干干净净清清爽爽的浏览网页吧。

6、其它推荐。

我有软件洁癖，所以喜欢各种简洁实用的软件，PC上听歌用网易云音乐，输入法用QQ输入法，安全软件用百度安全卫士，下载用QQ旋风。虽然这几个软件都不算主流，但胜在简洁够用，而且不会弹出任何广告。受不了酷我上的各种弹窗，搜狗输入法提示你该安装搜狗浏览器，360安全卫士不停提示你的电脑有漏洞，当然更不用说臃肿的迅雷，一个下载软件居然还会弹新闻。

安安静静工作不会主动打扰用户的软件才是好软件。

 

【读书笔记】HOW TO READ A BOOK

发表于 2014-02-10   |   分类于 读书旅行

《如何阅读一本书》

2014年的第一本书，真后悔这么晚才遇到她。

做任何事请都是有规则的，之前看过的《用户体验要素》讲的就是规则，互联网产品很复杂，但《用户体验要素》给了我一个规则让我按照这个规则去分析理解产品。同样，这本《如何阅读一本书》也是讲规则，给读者讲读书的规则。

阅读的目的

1、获取资讯；2、增加理解力。

第一点比较容易理解，就如同读报纸、杂志一样，看到的是新闻资讯，让我们知道这是什么，增加我们的知识面。而第二点就是本书的目的，通过阅读来增加理解力。有的书很容易读，有的书读起来却很困难。原因是我们与作者不相等的理解程度，无法理解作者传递给我们的思想。而我们所要做的就是通过自己的努力与阅读技巧把这种不相等的理解力控制在一定程度内，然后我们才能读懂作者想要表达的意思，只有这样我们才能把自己的理解力提升到与作者相同的水平。通过阅读我们自身成长了，这才是阅读的目的。

阅读的层次

1、基础阅读；2、检视阅读；3、分析阅读；4、主题阅读。

基础阅读就是基本的识字理解，看到一段话知道他在讲什么，这些基本上在小学就掌握了。就是我们小学初中大学语文考试时前面的基础题考察的内容。

检视阅读，也就是快速阅读，在很短的时间内了解到这本书在讲什么。这里面有分两种略读和粗读，略读就是随便翻翻，看看前面、目录，中间翻两页。粗读就是不追求深度思考，粗略的把书看完，看不懂的直接跳过。略读的重要性在于让你在很短的时间内判断这本书是不是你感兴趣符合你要求的书，现在市面上书籍那么多，你得判断一下那些书该读那些书该直接放弃。而粗读就是避免我们在小细节的难点上花太多功夫而无法继续，遇到难点直接跳过，看完全书再回来看难点会有更好的理解。

分析阅读，通过各种技巧把书读透，完全理解作者想要表达的意思，把自己的理解力提升到与作者同一个水平。分类、找关键字，划重点句子，判断主旨，同过这些来找到作者想要表达的真正意思。就是我们初中、高中做阅读题中做的内容。（PS：这里就要说坑爹了，上学的时候做的那么多阅读题，从来没有人告诉我做那些东西有什么样，看完这本书之后才知道然来都是为了增进阅读理解力的）。

主题阅读：打个比方，就是学论文的时候我们有一个选题，现在我们要看很多书，找出其中能支持我们观点的内容作为我们的论据。把我们找的所有论据总结起来形成一个独到的观点，写成一篇论文。

当然原书讲的远远不止这些。例如，实用书怎么读？

判断你是否读懂了一本实用书，就是在你认同作者的观点同事是否按照作者说的那样去做。《如何阅读一本书》就是一本实用书，我认同作者的观点，我想我也会按照他们的规则去阅读其它书。

2014年，读懂更多的书，提升自己的理解力。

 

 

我的1314

发表于 2014-01-29   |   分类于 个人提升

写点东西，纪念这个让我人生发生转变的一年。

整理这一年看过的书，自以为看过很多，结果不包括杂志居然才20多本，好吧，自认为是个热爱读书的人其实不过是在秀另外一种优越感而已。对我而言，阅读还只是在地铁上打发时间以及提升工作能力的工具，还没成为一个能独自享受的爱好。2014年，希望有更多的阅读、更多的思考和更多的分享，不再局限在自我狭小的世界里。

在朋友雷包子的帮助下，我发起的微信订阅号Day Day Up，从2013年10月7日起，坚持了快4个月，每天把好的文章或一些亲身感悟分享给渴望成长的人，做这个的目的就是为了展现自己的价值观，进而在茫茫人海中找到具有相同价值观的人。来年，希望能探索出更好的模式和更好的内容，也希望能通过这个平台来获取和展示更多的独立而新鲜的思想。

买了域名和空间，从6月份开始写博客，在浩瀚的网络上有了自己的小窝。以前写博客更多的是心情表达，而现在写博客更多的是经验总结与思考。我坚信失败了爬起来后不回头看看你掉下去的坑，下次你还是会掉下去，所以得时不时得回头看看，看看自己的脚印是否歪了。写作是我的一个梦想，2014年，我会继续写下去，总结自己的过失经验，继续提高工作能力，也锻炼自己文字表达的能力。

有机会有心情的时候就会换上裤子和鞋子围着小区跑步，一整天都窝在办公室里让人特别喜欢外面空气冰冷与新鲜的感觉。有力气的时候就多跑，没力气的时候就少跑甚至不跑，也正是由于过于随性，没有规划安排，以致今年身体状况明显不如从前。2014年要制定详细的计划，不再只是凭心情锻炼。怎么能让身体健康问题影响自己前进的脚步。

在我第一篇博客里我计划在一年之内成为一个入门级的产品经理，这个我做到了，顺利了成为了一名产品助理，由于某些原因，我成为了这个项目产品方面的临时负责人，这5个月我做成了第一款手机APP，第一款腾讯应用，虽然它们都不够优秀，甚至可以用垃圾来形容，但这个过程却让我成长很多。跟上一份工作一样，没有人教你怎么做，所有的事情只能靠自学，Axure画原型图，Mindmanager做思维导图，写PRD文档，学会了该用什么样的思维去分析产品，也了解一个产品从一个点子到成型所经历的所有流程。我知道我做得不够好，画的图没有别人的精细，写的文档没有别人的清楚，流程安排会让人抓狂。2014，我会继续加强我的职业能力和工作技能，这是我的工作也是我的梦想。

写到这里突然发觉，心情才是我这一年变化最大的地方，我不再一个人在那里琢磨小心思，我开始很放肆地笑，和大家在一起的时候我很合群，能为大家制造欢乐，成为大家眼中的二货。独处的时候我也能一个人很开心的逛街看电影，偶尔也有不爽时候，比如试衣服的时候旁边没人拿我脱下的衣服，看电影的时候整个电影院就我旁边座位是空的，买饮料的时候收银员对我说“第二杯半价”……但是这些都成为事后自我调侃的笑点，却丝毫不影响我以积极向上的态度来度过每一天，努力成为一个有趣的人。

元旦的时候没写这份总结，是因为总觉得只有农历新年才有除旧迎新的感觉。今天是农历年最后一个工作日，明天就要背着行李回家了，离开上海回湖北。总有以前的同学朋友问我会不会在上海待下去，在可预见的未来里我会在魔都一直待下去，喜不喜欢这里我不知道，但我知道这里能让我在不依靠任何人的情况下生存下去。

我是一个随便把我扔在哪里都能独自乐呵的人，下次，不知道生活又会把我扔在哪里？

 

【读书笔记】用户体验要素（未完待续）

发表于 2014-01-15   |   分类于 读书旅行

书名：《用户体验要素—以用户为中心的产品设计》

用户体验五要素：变现层、框架层、结构层、范围层、战略层。

设计时采用自下而上地建设。先确定最底层的要求再往上一层次走。

一、战略层

战略层需要确定好产品目标和用户需求。

产品目标又分为商业目标、品牌识别、成功标准。

商业目标是指产品上线好需要获得那些商业目的。是展示广告还是贩卖服务。成功标准是指产品上线后满足那些目标才算成功。

用户需求的获得有三种办法：用户细分、可用性分析和用户研究、创建人物角色。

用户细分是指把具有相同特征的用户划分在一个类别里再分析这个类别的用户有那些需求。而创建人物角色是指创建多个假想的目标用户，越具体越好，每做一个产品或功能，都从这个假想用户出发，看新的产出是否满足这些用户的需求。

2013年，犯的错误

发表于 2014-01-13   |   分类于 产品经理

做产品，不带经理，或许这句话能呈现我现在在公司的状态，挂着产品助理的Title，做着产品经理的工作。从2013年9月9日入职到现在，犯了诸多错误，总结一下，在以后的工作中进行改进。

错误一：责任心。

*为了满足编辑对于产品设计图的需求，反复要求设计去修改设计图。

责任心可能是一个老生常谈的话题，但作为PM，责任心的范围要光很多，我要对用户负责，满足他们的需求，我要对主编和推广负责，尽量满足他们的工作需要，我要对设计、技术负责，不能让他们做没有意义浪费时间的事，我还要对公司，对产品负责，让产品能够盈利能够支撑公司的发展规划。而且很多时候，对一部分人负责任就是对另一部分人不负责任，这时候就需要PM就要权衡得当，知道把谁的利益放在第一位。

错误二：流程。

*需求提交流程不规范，导致需求开发进度过慢。

当初没做PM的时候，对这个词甚是膜拜，因为每个招聘信息，每篇产品人的总结还有与产品人当面交流都会把这个拿出来说事，说什么产品经理要懂流程。刚开始不理解，而现在也开始慢慢开始懂流程，也了解了流程的意义。我们是一个团队，我们需要一个规范来使我们能够高效的工作，而流程就是这个规范。PM的工作就是创立这个流程，然后按让所有人按照照这个流程去工作。

错误三：规划。

*开发手机APP时没有做过详细的规划，导致应用开发完很长世间应用还没上架。

本来是写计划的，但是计划大多是指个人，而对于产品，我们要规划。我们什么时候完成原型和PRD，什么时候开始设计，什么时候开始开发，然后又是什么时候开始验收，这些都有计划，有开始时间，有结束时间，还应该有结束标志。大家都按照这个规划走。

错误四：沟通。

*一个人闷头做事情，没有与项目组其他人沟通，做了很多无用功。

产品是大家的，而不是某一个人。所以当对产品做出任何决策时应该告知大家修改的内容，听取大家对于修改的意见。然后再做决定。多沟通，了解每个人的想法，了解项目组内部人员的深层次需求。

错误四：用户。

*技能缺乏，时间不够，种种原因导致很少与用户接触。

用户是产品存活的关键，再好的产品不能满足用户的需求都是白费。而了解用户需求的唯一途径就是用户调查。问卷、网上沟通、当面交流、数据分析，这些行为都可以收集用户的需求，而这些工作我都没有去做。

错误四：单线程

*需要自己管理的产品太多，当沉浸在一个项目时忽略了其它项目的改进。

 

做产品经理而不是功能经理

发表于 2014-01-06   |   分类于 产品经理

这是我在微信朋友圈分享的一篇文章的标题，鬼脚七写的。担任项目负责人这么久也开始越来越觉得这句话的正确性。

今天有与推广闹得有点小不开心，因为她提的需求我没做。

在文章下面的分享按钮加上分享到微信朋友圈。我不做的原因：我们的网站还没有移动端，分享到朋友圈中的链接点击后打开的WEB端网页，基本上无法阅读。虽然修改分享代码基本不耗费多少时间，但我还是不愿意因为这个根本没有任何意义的修改而去麻烦技术。

之前我犯了一个错误，在屏幕下方加上一个可以关闭的大的注册导航条，画好原型，让设计做出设计图，让技术开发，完成后呈现在我面前我就发现这是一个多么大的错误，想注册的用户自然会找到入口，不想注册的用户即使点击了最后还是会流失的。加上后还影响本来的阅读体验。

没做一个功能都考虑清楚，这是否有必要，目的是什么，功能做好后是否能实现目的。另外不仅要推敲来自用户的需求，还要推敲来自项目内部的需求。

项目危机

发表于 2013-11-21   |   分类于 产品经理

手机APP危机处理。

1、问题：做分享功能，由于大家都是第一次做直接导致前期准备工作不足，功能完成的时候发现新浪和腾讯的高级接口没审批下来。

原因：前期调研不详细，做一个功能之前没有了解到需要准备的东西，没花时间去仔细研究。

解决办法：前期调研工作充分，不要光顾着往前冲，适当的时候听下来做点细致的研究工作。接下来的腾讯应用就要注意这个问题。

2、问题：开发完后测试有问题，修改后再测试又发现另外一个问题，反反复复改个不停。

原因：第一是因为我没有制定相应的测试规范。应该每个APK文件出来后让技术去做第一轮的测试，毕竟应用是技术改动的他们更清楚改动会造成什么问题。第二是技术部门功底不够，虽然我很不喜欢找别人的原因但在这个项目中这个问题明显的表现出来。反复的修改问题，截至日期延后之后，分享功能还没完成。

解决办法：产品经理应该制定完善的开发流程，反复的沟通测试尤其耽误时间，今后的开发中应先要求技术做一次完善的测试产品再进行审核。把测试任务放到技术部门做。另外产品经理只负责项目管理而不负责人员管理，所以当技术人员实力不够的时候应直接跟技术经理交流。

3、问题：当截至日期到了的时候功能还没做完。

原因：正如之前看过的文章里说的那样，没有一个项目是从容不迫的上线的，到该上线的时候你会发现项目出现很多之前没想到的问题。

解决办法：项目分期进行，V1.0版本只做最基本的功能，到下一版本在一个一个功能加，快速迭代。不要试图一次性把所有功能都完善。这次到项目上线的日期，分享功能还是不够完善，果断砍掉，留到下一个版本解决。

谈谈项目开发流程

发表于 2013-11-09   |   分类于 产品经理

安卓APP已经开发完成了，下周进行最后一轮的测试，安排好后续的工作流程，推进上线计划，顺利的话，下下周就可以在各大应用平台推出了。

应该是8月23号接到老大从西藏打来的电话，让我负责这个小项目的所有事情，从原型到文档，到项目跟进管理，到最终的测试和上线。想不到一个小小的安卓手机APP居然花了7周左右的时间。但毕竟这是我第一个全权负责的项目，得总结一下，好为以后的项目开展做准备。

按流程一步步细说。

1、立项

按照一般的立项过程，首先得做用户需求调查和市场分析，看看用户在心理APP方面有那些需求，看看市场上是否有类似的产品，进行对比分析，最终确定这个产品是不是应该做，应该怎么做。很遗憾，在这个小项目中，这个步骤被直接省去了。没有任何调查，只有一个长途电话，然后我就立马开工。没有所谓的BRD、MRD文档，没有任何用户调查。可能这只是一个小项目，所以很多东西都简化的，但在以后的产品中得把这一点做好，做好用户调查，做好需求分析，让每个产品都有它存在的意义。

2、画原型

这恐怕是每个产品经理最开始做的事情吧，作为一个产品的设计者，原型图需要的不是多好看，多保真，需要的是把需要表达展示的内容展示出来，需要把基本交互做得更符合用户的操作习惯。很高兴的是我一开始就意识到了这个问题，没有花大把的时间来弄原型图的外观设计，那是设计师的事情，产品经理把原型画得太好，太精细，反而会影响UI设计师的灵感。

3、写PRD文档

PRD文档即产品需求文档。就是把原型图用文字语言描述给设计和技术，遗憾的是这个我做得非常不好。找借口的原因是我根本没有任何PRD文档，也没有人教我如何去写。我自己的本身问题也是最终要得问题，我没有意识到PRD文档的重要性。人们不喜欢去思考，所以你需要用清晰明了的语言去向他们解释这些功能是如何实现的，这就是PRD文档的作用，详细描述每个功能是如何实现的，然后技术照着做就行。详细、清晰、明了，把看PRD文档的人当傻瓜，抱着这种信念去写PRD文档才能得到大家的认同，才能减少沟通的时间成本。

4、设计

设计师绝对是这个世界上最苦B的职业，因为世界上每个人对于美的理解都不一样，我喜欢蓝色，你喜欢绿色，她喜欢紫色，所以每个人都能够在设计师的设计上指指点点，就像这次APP的设计，为了配合心理的文章内容，设计在配色上选了比较单调的颜色，结果因为老板喜欢红色，所以推翻重做改为红色。而作为一个不懂设计的项目负责人，我需要做的就是信任。给予设计足够的信任让她们自由发挥。第一个是因为我不懂，第二个则是我坚信每个人都有自己擅长的东西，设计擅长的就是设计出用户喜欢的色彩和布局，那为什么不去信任她们呢。

5、开发

产品经理得拿刀逼着开发做事。这个是我做这个项目的体验。没办法，开发资源紧张，所以你得跟他们定好进度，约定什么时候做完，提前去追问项目进度，在约定的时间之前一定得逼着开发把事情做完。之前的老大就是因为这个被老板开了。

6、细节

项目进展过程中总会发生你想想不到的问题。而这时候需要做的就是找到相关的负责人，跟其商量找到最合理，最迅速的解决办法。

 

先就到这里了，等APP测试完上线后可能会有新的体会，到时候再添加进去。
