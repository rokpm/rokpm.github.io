不做画图小子

发表于 2014-11-16   |   分类于 产品经理

入行已经一年多了，大大小小都算上也经历了5个产品。也逐渐建立了一个知识体系，试着把自己总结出来的经验分享出来。

产品经理不是画图小子，原型和交互细节不是最重要的事情。

我一直在犯这样的错误，接到一个产品任务时，简单的列下功能的思维导图就开始画原型，接着盯在原型的细节上，纠结于产品的交互体验。经常画原型到深夜，还在微博上炫耀，以为给用户呈现完美的使用体验是产品经理最应该做的事情。

慢慢的理解到，在产品初期，原型和交互体验不重要，当然不重要并不意味着随便弄弄，只是它的优先级比较低。

交互体验不好，页面设置不合理，按钮位置反人类，小事，更新一个版本就行；UI界面不好看，没事，多招个设计，3天就能出一套方案。对于产品初期，这些都是小问题。

产品初期最应该注意的问题是尽早地去验证产品是否解决了用户疼点。

根据《精益创业》中总结的用户增长曲线图，初期用户只有很小的一部分，而且这部分用户喜欢尝试新鲜的事物。再通过小范围的去灰度测试，将初期用户控制在一定数量，前期该担心的不是产品是否有完美的交互，而应该去考虑产品是否解决了用户的疼点，真的解决了用户的疼点，体验只要不垃圾到无以复加的地方，用户都会去忍受，而且这是一个很好的验证是否满意需求的办法。体验稍微差点用户就走了，说明….你的产品根本没有解决用户的真实的疼点。

如果我开始创业

发表于 2014-10-20   |   分类于 产品经理
如果我开始创业，我不会立马开始创业。

我会去了解自己所从事的行业，深入的与这个行业的从业者去接触，充实自己的行业知识，成为这个行业的专家，不去做想当然的做事情。


如果我开始创业，我会专注自己的创业。

我会瞄准自己的目标不会松手，别人融了多少资金，IPO市值多少，或者什么行业又站在了风口等待猪往上爬，这些都与我无关，我只关注我所在的行业。


如果我开始创业，我会大胆的说出自己的想法。

我不会遮遮掩掩害怕别人抄袭了我的想法，那只能说明我对这个行业还了解得不算深入，说明我不够牛逼，我会汇聚来自四方的意见来完善自己的思路。


如果我开始创业，我做的第一事是组建团队。

我会寻求靠谱的队友，组建完善的团队。聚集一批向同一个方向努力的人，我会与他们分享创业所带来的艰辛已经福利。


如果我开始创业，我会沉淀下去。

我不会张口O2O闭口B2C，我不会去追逐那些假大空的概念，专心做好服务于用户的事情。同时我也会去阅读与沟通，完善自己作为创业者的能力。

失败经验：下次，我会如何去处理需求

发表于 2014-09-28   |   分类于 产品经理

月初接了一个活动方案，到月末了依旧没拿出能够通过的方案。
为什么我做不好我的工作下次，我应该如何去做。


1、去理解老板的想法
小公司，需求都是由老板决定的，拿到了需求我就开始按照自己粗浅理解去做，草草的画出原型图然后就给总监进行评审，做出来的东西当然不能让人满意。
我不是一个画图的！我曾经这样对老板说过，但到了做事情的时候，我却总这样去定义自己，拿到需求立马开始动手画原型，缺少分析过程，缺少验证过程。
首先我应该去理解、分析需求，老板说我要什么什么，这个要怎么怎么样，我应该去理解老板的目的即为什么要这个，目的是什么，而不是老板怎么说我就怎么做。

2、如何去做？
开始画图了么？不，等等，你要结构化你的思维，也就是说你得先把自己的思路整理清楚，在大脑之中行程整个框架并用思维导图表现出来的，这个需求涉及到那些方面，优先级又是怎样的。

3、评审、评审，让别人帮助自己
对！产品内部对需求的功能结构进行评审。大家都是做产品的，所以拿出脑图，大家都明白你要做什么，这时候去评审，然后大家一起对脑图进行修改，补充以及完善

4、原型其实不重要
这个部分就没有什么可以讲的了，根据思维脑图，把每个页面用axure表现出来，并整理成流程图。对了，在这里提醒一句，真心没有必要去用Axure来做那么复杂的交互，太耽误时间而且无用。整理出能让设计和技术看懂的流程图就OK了。

5、整理成PPT去让老板拍板
总监说，写PPT是给老板看的，让老板看完PPT可以直接拍板决定做还是不做。
PPT中对整个方案进行说明，列出主要流程，对一些重要的细节进行说明，以及后续例如运营的一个简单思路
老板看完之后能够立马清楚这个方案是是什么情况，能不能达到我想要的效果，从而决定做与不做。

好了，以上是我从失败中总结出来的经验，分享给大家，当然每个公司的流程不一样可能处理方式也不一样，仅供参考。也希望你能说出自己的想法，共同学习，共同进步~~~

最小化产品

发表于 2014-09-17   |   分类于 产品经理

最近在看《有的放矢：NISI创业指南》，现如今市场速度太快，应该尽早的产出产品市场化，可以快速验证想法，进而根据用户的意见来进行修改。

昨天在知乎上看到了一个失败的创始人在创业失败后写的一篇文章《入门错误-过早产品化》链接：http://zhuanlan.zhihu.com/denny/19698570

提到了更快速的产品验证方法，创业初期，对产品以及市场还没有清晰的了解时首先应该做的是快速验证想法是否可行，而开发一个产品至少要耗费2个月时间，更糟糕的是产品开发完成投入市场之后才发现没有用户为之买单。

也想文章中提到的那样，如果要做一个电商产品，首先应该做的是去淘宝开店，甚至更轻就微博和微信去推销自己的产品，看是否有用户买单。

我们的产品也是如此，淘宝基本上可以做到100%的基础功能，为何我们不现在淘宝上验证下我们的想法，看用户是否买单，这样做也能尽快去修正我们前进的方向。

从需求到产品

发表于 2014-09-13   |   分类于 产品经理

周末游，来到了惠州巽寮湾，路上两件事让我不经细想了下。
1，租车出游，路上听广播，发现广播里的歌曲都是一些老歌;
2，路上经过一段路，发现两三百米，左右两边都是卖游泳圈，游泳裤的。所有的摊位基本上一模一样。

人群，需求

为什么广播里的歌都是老歌？

由用户群体决定。
对于什么样的用户群体喜欢听广播？
答案显而易见，司机！
由于视线被驾驶占用，所以司机只有通过听广播来打发无聊的驾驶时间。
而现阶段，开车的司机大部分是出租车司机和买的起车的人，那么可想而知，用户群的年龄大概是多少。
所以对于我这种90后而言，广播里都是放着老歌。

需求我确定了，那怎么去满足需求。

人们去海边玩，需求产生了，人们需要游泳圈，泳衣。于是四五十个商家都在提供相应的服务来满足用户的需求。如果我是一个商家，我如何从中突围。

提供独特的服务。其实除了必备的东西人们还需要其他的。比如防水的手机套，我问了几家才找到，当然还有更多的需求等待挖掘,例如提供简易的海钓工具等，总有一些需求是其它商家没有提供的。

最好的产品是提供给用户独特的价值。

你需要的，而且只有我这里能提供，你不找我找谁？

新公司，新项目，第一个任务就没有完成。写产品立项文档，一向自诩思维清晰，想法众多，这次却有种脑袋被掏空的感觉。

我想是因为我自己没有理解这个项目，是他们没说清楚？还是我没想清楚？我真心弄不懂。我的思维被局限住了，我完全没有找到令我兴奋的亮点。

如何去理解这个项目？如何去理解别人的想法？是我目前遇到的最大问题，所以我现在最重要的任何就是解决这两个问题。

产品小结（广州）

发表于 2014-07-08   |   分类于 产品经理
已经做产品快一年了。


做了一个简单的腾讯应用，跟技术折腾了好几个月做了一款手机APP，ios版最后还被APP STORE拒了，又折腾了很久在6月中旬把主站改版上线了，有时候真心不知道自己算是成功还是失败。


自己觉得成功的地方在于在没有任何人带的情况顶住了来自同事的压力，以及自己能力不足带来的焦虑，一个人支撑了整个项目的产品工作。而失败在于做的三个产品都算不上成功，只能说做完了。



没人指导是我遇到的最大的问题。这样导致即使我做错了也没有人在旁边跟我说，于是我按照错误的思路一致往下做，等我发现自己错了的时候时间已经晚了，不可能再修改。自己和技术都要开始忙下一个产品，根本不可能再回去弥补我当初犯下的错误。


什么是正确的产品？什么是正确的产品经理？什么是产品经理正确地思考方式？什么又是产品经理正确的做事方式？

做得越久我发现我的迷茫度就越高。我总觉得不是现在这样，但我又不知道应该是什么样的。


我觉得产品就像一篇小说一样，做产品就像写小说。并不是一开始就就动笔写，首先我需要主旨，也就是我写这篇文章要表达什么，我还要积累很多相关的素材，我还要去准备笔和纸。最后我开始写，当我开始写的时候，这个故事肯定已经在我心中，我只是用笔把它誊写下来而已。做产品也是如此，当我开始画原型的时候肯定大部分准备工作都做好了，当我真正开始画的时候整个流程包括核心概念肯定已经烂熟在心了。


这是我想的，可能也是理想化的。我不知道什么是正确，什么是错误，所以我一直都在探索。

如何高效地处理需求

发表于 2014-05-15   |   分类于 产品经理

前段时间的工作被我弄得一团糟，估计整个项目组，技术、设计、推广还有编辑都对我有很大意见。

之前的流程是，编辑部门和推广部门提出需求，我个人稍加评审后通过管理工具推到设计和技术那里。设计和技术不停地对需求提出意见，编辑和推广又不停的催促为什么产品还没完成，于是我卡在中间左右为难。


上次，编辑部门说我们的搜索功能太弱了，如果能接入百度的搜索会更好，于是我去了解了百度的站内搜索，通过百度的工具建立了框架，让技术去做。到技术开始做的时候，觉得百度搜索的UI风格与我们网站不符，而且很容易导流到其它网站上去，造成用户的流失。


出现这种情况的原因有很多。



1. 我个人不强势，无法说服需求提交人放弃暂时不合理的需求，也无法说服开发部门对需求所提的意见。
2. 需求提交人对所提的需求没有思考清楚，只抛出一个需求给我，我们需要什么什么，你去做一下。而对需求是否合理不加以判别。
3. 需求开发部门没有了解到关于需求的详细信息，从自己的角度判断需求不合理，然后就不愿意花费时间去做。导致需求长时间处于等待处理的状态。

针对出现的问题，我设计了一套需求提交的流程：





1. 需求提交人员必须在提交需求的时候填写需求申请单，帮助自己捋清楚思路判断自己所提需求是否合理，也帮助产品能顺着需求提交人的思路去思考需求。
2. 产品对需求提交人的需求进行整理，初步剔除不合理的需求。
3. 每周五召开需求评审会议，要求需求相关人员：需求提交人、需求开发人员参加，由需求提交人对所提需求进行评述，评审后得出评审结果。
4. 对于可以立马开始做的，产品、设计、技术安排时间进行处理。对于将来做的，设定启动条件，当条件满足后，再对需求进行处理。对于不合理的需求则放弃进行下一步。

以上是我对需求处理的基本思路。


这样做的好处是：



1. 减少沟通成本，不用产品卡在需求提交人和需求开发人中间传达彼此的观点，直接把他们拉在一起直接沟通来解决问题；
2. 提炼需求，通过需求单让需求提交人能够对需求进行深层次的思考，通过评审让大家对需求进行讨论，有助于放弃无意义的需求；
3. 不干扰技术的工作，通过整个流程，每周五集中对本周所提需求进行评审，下周一产品将上周所有需求提交到需求开发部门，中间的时间技术部门就不会因为零零散散的需求而干扰主要工作的进度。
4. 减少产品的压力，评审会与上给出需求的处理方式以及完成时间，需求提交人就不会时不时的去催促需求的完成情况。

时间管理

发表于 2014-04-17   |   分类于 个人提升

最近工作效率有点低，于是又重新把大学里学来的那套时间管理的方法重新捡起来。

接触到时间管理是偶然在大学图书馆看到的一本书《小强升职记》，用故事的方式讲解了时间管理的各个概念并运用于实践，忘了差不多了，但还好主干知识还记得。

第一个是GTD，第二个是番茄工作法，两个搭配起来就是工作神器了。

GTD的全称是get things done，这也是一本书的名字，买了，但外国人写的东西当时觉得太晦涩难懂了于是送人了。

首先来说说GTD的流程：

第一步：收集（inbox）

清空大脑，大脑是用来思考而不是记忆的。把所有要做的事情从头脑中拿出来装进inbox中。这样就不用去考虑我有什么事没做时带来的焦虑。

第二步：分解

把每一项inbox中的任务分解为若干个可以立即执行的下一步。举个例子，我要构思手机APP的原型。那么下一步行动就是：

1、收集市场上存在的竞品APP；

2、试玩每个APP；

3、为每个APP画思维导图整理思路；

4、通过几张思维导图以及自己的资源整理出我们的手机APP的脑图；

5、安排手机APP的功能结构；

6、用纸笔画草图；

7、用AXURE画低保真的原型图。

一个大的任务就分解为6个小任务，每个小任务具有很强的可执行性。

第三步：安排

讲上述分解的下一步行动安排在日程表中，选择执行时间。

以上就是简要的GTD流程，理一下你可能就清楚了GTD的精髓是什么。GTD就是能够解放我们的大脑，让我们的集中精力去解决问题，而把思考问题放在纸和笔上解决。

顺便介绍下我的GTD工具，市面上所有的GTD工具基本上都试用过，觉得最好的还是Doit.im。手机、PC能同步，大多数应用只有手机端，但因为工作性质大部分工作时间多对着电脑，所以PC端的输入以及操作更加快捷。

用完GTD那为什么我们还需要番茄时钟？

上图就是番茄时钟，国外做饭时用来定时的，比如这汤要炖45分钟，把番茄时钟定好45分钟后去做其他事情，等时钟走完闹铃后再回来关火。

番茄时间工作法的流程很简单，从下一步行动中拎出一个任务，预计要耗时50分钟。我们定25分钟为一个番茄时间，定好番茄时钟后开始做事，等时间走完就休息5分钟，然后再定一个番茄时间，再继续工作。25分钟事情做完。耗时2个番茄时间。

也许你会觉得这样做毫无意义。
