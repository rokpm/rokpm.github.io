老板最看重的能力

发表于 2015-07-31   |   分类于 产品经理

作为一名员工，老板最看重的能力是什么？

我的答案是解决问题的能力。作为非管理层的我们，老板招我们进来不是为了我们来制定公司战略层的方案，而是为了执行、实现老板战略层的目标。而在执行过程中，会遇到无数见过没见过的问题，这时候解决问题的能力显得尤为重要。

在工作、看书、交流的过程中，我形成了一套解决问题的方法论，

发现问题（Observe Problem）：在翻译“发现”时我本来想用find，但感觉find在传递一种碰巧的感觉，而这里的发现问题不是碰巧，需要有强大的知识储备，因为之所以能发现问题，是因为你知道正确的是什么，如果连正确是什么都不知道，那谈何发现。

解构问题（Deconstruct Problem）：这一步应该是最重要的，发现的问题往往是表层现象，如一座冰山，显露在水面的总是一小部分，而解构问题正是找到水下那部分真正重要并且重大的问题。

制定方案（Make Plan）：为问题寻找合适的解决方案，这里给2个小技巧，第一是不要过分追求完美，第二是思考的过程中尽量增加思考的频率而不是长度。

行动（Action）：大部分逻辑中行动是最重要，但在这个过程中，行动反而不是太重要，按照上一步制定的方案，每天推进，推进到不能推进为止。

总结反思（Reflective）：这一点是最容易忽略了，我们的工作都是在不停的解决问题，但解决完了却很好有人去反思，我发现的问题对不对，我的问题分拆的对不对，是否有更好的解决方案，如果有更多的资源我会怎么去做，行动中是否有做的不好的地方。这些都需要我们去总结，这样才能极高效率的提升我们解决问题的能力。

最近在经历第二次创业项目，创业过程中真是没有一天不会遇到问题，不会遇到风险。而这些问题、风险就是我们游戏中的怪物、BOSS，打败它们我们才会升级。

产品经理进化论

发表于 2015-06-29   |   分类于 产品经理

2年自我成长的产品经理经历，让我遇到了瓶颈而无法继续成长，于是约了在行上的产品大牛，在咖啡店聊了2个多小时，疑惑慢慢有了一些解答。

产品经理对于团队的意义（输出正确的内容）

产品经理对于开发团队最重要的意义是他是团队的眼睛和大脑，是团队中唯一对外接触的职位。技术的责任是功能是否很好的实现，设计的责任是产品的视觉效果是否美观。他们对外界并不了解，而这些对行业、竞品、用户的了解程度决定了产品经理对于产品价值主张的把握。
只有对产品价值主张的正确把握，才能保证产品经理输出正确的需求、功能、流程、原型以及版本规划等内容。所以产品经理需要花80%的时间用于输入上，20%的时间用于输出。

产品经理的知识范围（正确的输出内容）

如同网络游戏中练级一样，产品经理也需要掌握一些技能：
1.专业性

  • 开发：产品经理不需要开发功能，但需要了解功能的可行性以及实现成本；
  • 设计：原型设计中，产品经理需要了解各平台（WEB/iOS/Android）的设计规范；
  • 测试：保证技术、设计正确的输出；
  • 交互：为用户提供良好的操作体验；
  • 用研：定性分析；
  • 数据：定量分析；

  • 竞品分析：不是简单的分析竞品的外在功能，而是通过对竞品内在的分析了解竞品的价值主张。
    2.用户感

  • 用户行为：当前用户外在表现行为；

  • 用户动机：用户行为产生的动机；

  • 用户流向：当前用户的解决方案；
    3.leadership

  • 沟通

  • 资源控制



 

消费的幸福感总是很弱

发表于 2015-06-07   |   分类于 个人提升

在我的奇妙清单里，有一个列表叫做“想买的东西”，里面收集了我想买的东西，因为我总是觉得拥有自己想要的东西是一件幸福的事情。

抱着这样的一个心态，我买了很多东西，森海塞尔的耳机，喜欢的单肩包，宜家的相框和小桌子。我以为拥有它们我会比以前更加幸福，但现实却是相反的，拥有这些东西反而让我的生活更加复杂、烦躁。

我一直想拥有头戴式的耳机，于是海淘了一个，幸福感没有增强，烦恼却增加了许多。等待收货的15天里不停的刷ebay、转运网站、EMS，查看物流状态。耳机到了之后开始上网搜无损格式的音乐，iPhone不支持APE格式的音乐播放，于是去找无损播放器……一副耳机牵出了一堆需要我去处理的事情，细想发现，耳机带给我的烦恼远大于带给我的幸福感。

但是有些地方的消费却让我有很强的幸福感，租一间还不错的房子，换了iPhone居然都用了快2年了，付费的VPN，付费的主机空间。

能增强我幸福感的事物大部分都简化我生活，而非使之复杂化。

性价比高的都是垃圾

性价比高的都是垃圾，之前一直听说，但现在才觉得这句话的正确性。

因为贪便宜，放弃了一直用的主机服务商，转向淘宝，花了38元买了一年的香港主机，前三个月一直很正常，最近一段时间不停的更换IP导致无法访问，最后干脆停止运行，去找卖家咨询，客服不在线，淘宝上所有商品都删除，投诉无门。更加悲剧的是数据库没有提供导出功能，花了2天咨询几个技术同事才弄好。要是无法导出，我这几年写的博客就全部丢失了。


对了，还有淘宝上买的iPhone数据线，10块钱不到，外表看起来挺结实，我还帮朋友买了2根，之前也一直没问题，但最近开始无法连续充电，帮朋友买的2根也是如此。


一分钱一分货，很多时候确实如此。主机无法访问还好，数据丢了怎么办？数据线坏了也好说，但临时身边没有其它数据线就悲剧了。

当然任何事情都不是绝对，便宜的一定就很差，只是下次做消费决策的时候多想想。

那些年我掉过的那些坑

发表于 2015-05-27   |   分类于 个人提升

如果你去问一个产品经理，该如何更快成长的时候，80%情况下，他会说掉的坑多了就知道了。冲进一个已处于红海的市场，产品方向错误，对线下的流程控制不到位等等，但很少有人去写我们日常工作中的那些小坑。那我就来写写我犯了无数错误后总结的一点小经验。

1、不要写错别字。我曾经不止一次的将“登录”打成“登陆”，将“客户端”打成“客服端”。虽然只是小细节，但也反应了这个人的没有精益求精的工作态度。

2、积极、积极再积极。工作中不要做需要人照顾的Baby，不要等着别人来安排工作，自己主动去找事情做，别怕丢脸。

3、认真对待每一件小事。有些人说这是为了给领导一个好的印象，而我认为这是在减少麻烦，把那些小事一次性处理完美，之后就不用再返工耽误时间。

4、管理好自己的时间。学习一些时间管理的技巧，这能够极大的提高效率，让自己有更多的时间享受工作之外的生活。

5、注意沟通，别埋头做事。积极向领导汇报工作，不是为了表现自己，而是为了保证信息同步才能更好的合作。

6、把同事当做朋友。每天至少8小时，你都和同事待在一起，所以为什么不把他们发展成为自己的朋友呢。这样工作沟通会减少很多阻碍。

7、别太把自己当回事。这是大部分产品经理犯的错误。以为自己的职位是产品经理就真的是经理了，其实5年以内只是一个产品策划。要专注于自己的本职工作，不要老是去考虑公司的战略问题。眼界、积累没到位，就真的没有这个资格和能力。

8、注意自己的身体健康，没有什么比这更重要。

花了三晚上，第一次尝试用思维导图的方式重新去梳理一本书的构架，果然又重新学到了很多东西，捕捉到了很多之前被忽略的点。

刚开始进入产品经理的行业时，喜欢看《人人都是产品经理》、《结网》、《启示录》、《DON’T MAKE ME THINK》这类书，因为它们指导我怎么样去做产品经理的工作，它们是入门书，而慢慢地当我已经能够胜任产品经理的日常工作时，我在想怎样去做成一件事，失败的产品不仅是时间、金钱、精力、信心的损失，更是对智力的一种浪费。

记得之前看《How to read a book》，讲到读书的目的有两种，一种是获取资讯，增长知识面；一种是增加对事物的理解能力。我想这本书做到的第二点，成功可以复制这种笑话当然是欺骗小朋友的，但是这本书却是让我知道了应该按照什么样的方式怎样去创业，什么样是正确的，什么样是错误的。

发现问题是很多人都具备的能力，而解决问题的能力却很稀缺。当你发现自己具备第一种能力的时候，就应该花时间去培养第二种能力。

产品经理，你会什么？

发表于 2015-03-22   |   分类于 产品经理

我常在想，作为一个1.5岁的产品经理，我会什么？

知乎上有个笑话，为什么哪么多应届毕业生想做产品经理？答，因为他们什么都不会啊。再去看看各大招聘网站上产品经理的招聘要求，你会发现很多都没有对“硬技能”的要求。如果你面试一份技术的工作，面试官会让你写下代码看看你的编程能力，面试一份设计的工作，面试官会让你现场做一份设计稿看看你的设计功底，但是如果你面试一份产品经理的工作？好吧，基本上不会有考察技能的笔试题。

产品经理，你会什么？进一步去想应该是产品经理的核心竞争力是什么？

工具的使用？AXURE、XMIND、VISIO，但是工具往往是最容易学的，不要一个星期，axure的动态模板就用得滚瓜烂熟，而且画得再好，也没有一个交互设计师更懂原型的设计。了解互联网行业趋势？NO，看一个月的36Kr、极客公园，保证你张口就是需求、营销、站在风口上的猪、连接一切等等术语。了解开发流程？经历过项目的人谁不知道，然后再抖几句精益创业、敏捷开发中的内容，逼格又上升了一个档次。

如果不是这些，那么我们的哪些能力是能够给公司提供价值的？

对，我们确实做了一些公司其它人没有做的事情。我们通过自己的经验去分析来自内部或者外部的需求是否靠谱，将这些需求转化成设计、技术能够看得懂的可行性方案，再通过一些有效方法去推动大家一起去将这些需求转化为实际可用的产品。但是这些只是其它人因为时间、职责范围没有做的事情，也就是说他们只要想做也可以去做，只是熟练程度而已。

那么还是那个问题，作为产品经理，你会什么？你的核心竞争力是什么？你的哪些技能能够保证你不会被轻易取代？

按照产品的逻辑方式，一篇文章应该是发现现象，提出问题，分析问题，最后再解答问题，很抱歉，这里没有最后一步。自己经验尚浅，还不能完美的去解答，再就是觉得不存在传说中的终极完美答案，只能有待我们在成长的路上慢慢探索吧。

既然你在简书上看到这篇文章，那么很大可能性你和我一样，手机里N多阅读类的APP，简书、一个、知乎日报；微信里订阅了N多公众账号，罗辑思维、战隼、小道消息肯定少不了；需要看的网站太多，你不想一个个的看，于是捡起了RSS这杆利器，订阅了无数的网站博客，36Kr，极客公园等等等等；起床后，睡觉前，马桶上，你不会忘记刷刷知乎，为某个精彩的回答点个赞，抑或在简书上收藏一篇好的文章并分享到你的朋友圈。

很多时候我们都一样，对知识的渴求，对成长的急切，害怕错过优秀的内容。

但是，你还记得昨天看了什么吗？

我明明看过，分享过，甚至还保存到EVERNOTE，但我怎么也想不起我昨天看了什么。有时候能记起那么一丁点，而更多的时候是什么都不记得了。那么昨天的阅读对我而言有何意义，它们让我的脑袋肿胀，让我的双眼布满血丝，但是我却什么都不记得了。我甚至觉得他们还不如一部喜剧，至少一部喜剧能带给我2个小时轻松的心情。

我们看的是资讯，资讯只是增长了我们的见识，而成长则需要知识的积累与经验的总结。

这个信息爆炸的时代，我们从来都不欠缺资讯，我们缺的是把我们看到的提炼总结为自己的知识，缺的是把知识应用于实际的行动力。

初级敏捷开发一周总结

发表于 2015-02-01   |   分类于 产品经理

上周在老大的发起下开始用敏捷的方法来管理项目，之所以加个初级，是因为对于敏捷开发的理解只来源网上的几篇文章而已，并没有系统的去学习整个流程。

先简单地说下我们的项目是如何进行初级敏捷开发的。


第一步：整理目前的工作任务，将每个工作任务分为Task、Doing、Check、Done，并贴在看板上。

这一步的目的在于让项目成员知道我们做过什么，正在做什么以及未来会做什么，心中对整个项目有一个总体的了解，避免空想导致任务任务太多而无法处理。



第二步：将每个任务标明优先级、对接人以及完成时间。优先解决优先级高的任务，避免花费大量时间处理优先级低的任务。

并不是每个人都有良好的时间管理的概念和方法，而这一步就在于帮助项目人员做时间管理。作为产品，我也知道我下一步该整理哪些需求，出哪些文档。


第三步：每日晨会，说明三个问题1、我昨天做了什么？2、我遇到什么问题？3、我今天会做什么？

产品经理的一个很重要的任务就是保证团队的信息同步，请原谅我最近才想明白这个道理，而晨会的作用就是保证团队成员的信息同步，彼此知道其它人的任务进度，必要时能够腾出时间帮助那些需要帮助的同事。于此同时，产品上的改动和新需求，应该在早会后及时与所有相关的人员进行说明、讲解、讨论。


第四步：每天结束前产品、设计、测试处理Check中的任务

检查技术已经开发完成的任务，解决其中的BUG，大家一起，尽早将check中的任务放到Done列表中。



回顾整个流程，这套方法跟时间管理中的GTD以及腾讯内部的工作方法PDCA有些共通点，Plan、Do、Check、Action。只不过GTD是个人的时间管理，而敏捷是团队的时间管理。


在执行过程中，依然发现还是有一些问题需要解决。


问题1：无法帮助技术创造unbroken time。

由于反馈人员的缺乏，无法在技术把任务放到check栏中，及时给予技术测试的反馈，导致技术同时在处理Check和Doing中的任务。


解决办法：

在不增加人手的情况下，产品出产品文档的时候，在文挡后面附上简单的测试用例，技术开发完成后自己检查自己写的程序，只有通过所有的测试用例后才能将任务从Doingk标记为Check，而且自己检查也能加快BUG的解决效率。


问题2：每个人的任务互相交错

往往会出现的情况是，一个技术的任务需要另外一个技术提供接口，导致当前任务被卡住无法完成。


解决办法：拆分任务，将每个任务都拆分成可以单独执行并完成的任务，例如将问题中的任务拆分为两个，任务1，提供接口，任务2，技术开发。只有当任务1完成后才开始进行任务2.
